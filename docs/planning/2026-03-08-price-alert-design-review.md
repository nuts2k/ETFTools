# 到价提醒设计文档评审与修订清单

**日期**: 2026-03-08
**状态**: 待处理
**关联文档**: `docs/design/2026-03-05-price-alert-design.md`
**目的**: 在正式实现到价提醒功能前，汇总当前设计中的明显缺陷、矛盾点与建议修订项，作为后续设计修订和实施前检查依据。

---

## 一、结论摘要

当前的到价提醒设计方向总体正确：

- 功能定位清晰，区分了“信号告警”和“用户主动设置目标价提醒”
- 第一版范围总体克制，避免了过度抽象
- 已提前考虑调度器复用、收盘补检、自动清理等关键实现点

但该设计**不建议直接进入实现**。当前至少存在以下四类关键问题：

1. **触发语义不清**：文档使用“穿越/上穿/下穿”表述，但算法实际上判断的是“当前已达到目标侧”
2. **并发幂等方案不足**：`is_triggered` 不能天然保证多任务并发下只通知一次
3. **收盘补检可靠性不足**：15:01 获取收盘价、交易日判断、节假日处理都还不够稳
4. **规则定义前后不一致**：watchlist 约束、Telegram 配置要求、API 行为存在文档内部冲突

建议先修订设计文档，再开始编码实现。

---

## 二、必须优先修订的问题（Critical / Important）

### 1. 触发语义与判断逻辑不一致（Critical）

#### 现状

设计文档中多处将功能描述为“价格穿越阈值”“上穿/下穿目标价”：

- `docs/design/2026-03-05-price-alert-design.md` 第 1.1 节：价格穿越用户指定阈值
- 第 3.1 节 `PriceAlertDirection` 注释：`ABOVE = "above"  # 价格上穿目标价`、`BELOW = "below"  # 价格下穿目标价`
- 通知消息文案中也使用“跌破 / 突破”

但 `_should_trigger()` 的设计是：

- `below`: `current_price <= target_price + EPSILON`
- `above`: `current_price >= target_price - EPSILON`

这实际上表示“当前价格已经在目标价另一侧”，而不是“从一侧跨越到另一侧”。

#### 风险

这会导致以下行为与用户直觉不一致：

- 用户创建提醒时，若当前价已经满足条件，下一次检查会立即触发
- 用户重新启用已触发提醒时，若当前价仍满足条件，会再次立即触发
- 用户手动切换方向后，也可能出现“刚保存就下一轮立刻触发”的情况

#### 修订建议

必须在设计层明确采用以下其中一种语义：

**方案 A：到价提醒（推荐）**
- 文案统一改为“到达目标价 / 达到目标价”
- 不再使用“穿越 / 上穿 / 下穿”作为核心定义
- 创建或重新启用时，若当前价格已满足条件，则：
  - 要么禁止保存并提示“当前价格已满足该提醒条件”
  - 要么允许保存但提示“下次检查将立即触发”
- 为避免体验歧义，更推荐直接禁止“已满足条件”的创建和重新启用

**方案 B：穿越提醒**
- 保留“穿越 / 突破 / 跌破”定义
- 需要新增“基准侧别”字段或“上次检查价格侧别”逻辑
- 仅当价格从阈值一侧跨越到另一侧时才触发

#### 建议结论

第一版更适合采用**方案 A：到价提醒**，实现简单、语义稳定，也更符合用户“到这个价提醒我”的日常表达。

### 2. 并发触发与幂等设计不足（Critical）

#### 现状

当前设计中的检查流程是：

1. 查询所有 `is_triggered = false` 的提醒
2. 在内存中逐条判断并设置为已触发
3. `session.commit()`
4. 按用户分组发送通知

文档边界情况中写道：

> 并发触发（盘中 + 收盘补检同时触发同一提醒） | `is_triggered` 字段保证幂等

该说法不成立。

#### 风险

在以下场景中可能产生重复通知：

- 盘中检查和 15:01 收盘补检同时扫描同一条活跃提醒
- 手动触发接口与定时任务同时运行
- 未来部署为多实例/多进程时，多个执行器同时读取 `is_triggered = false`

两个执行器都可能在“查询阶段”看到同一条提醒尚未触发，然后分别发送通知，造成重复推送。

#### 修订建议

文档中需要明确最小并发保护方案：

1. **调度器层面避免重入**
   - 单实例部署下，为相关任务设置互斥执行
   - 若使用 APScheduler，需要明确 `max_instances=1` / `coalesce` 等策略

2. **数据库层面原子占有提醒**
   - 不能只靠“先查后改”
   - 应改为“条件更新”或等价原子方案：只有真正从 `is_triggered=false` 成功更新为 `true` 的执行器，才允许发送通知

3. **重新启用和数量限制也要考虑竞争条件**
   - 若多个请求同时创建提醒，单纯先 `count` 再 `insert` 不能严格保证 20 条上限
   - 第一版若接受“弱一致”，文档中应明确说明；若要求严格限制，则需额外约束

#### 建议结论

第一版至少要把“**只有成功原子占有提醒的执行器才能发通知**”写进设计，否则会出现重复提醒。

### 3. 15:01 收盘补检与交易日判断不够可靠（Critical）

#### 现状

设计文档默认认为：

- 15:01 时数据源通常已返回收盘价
- 只要 `day_of_week="mon-fri"` 就足以限制非交易日
- 节假日第一版不处理，价格不变，不会误触发

其中有一部分只是经验判断，不足以形成可靠的系统约束。

#### 风险

- 某些数据源在 15:01 返回的仍可能是未最终确认数据、旧值或延迟值
- `mon-fri` 无法正确表示 A 股交易日（法定节假日、调休等都可能偏差）
- 若非交易日返回旧价格，基于“当前价格 <= / >= 目标价”的算法仍存在误触发风险

#### 修订建议

设计文档应补充以下最小可靠性要求：

1. **交易日判断**
   - 优先基于交易日历判断
   - 若暂时没有交易日历能力，需在文档中明确 `mon-fri` 只是临时降级方案，不可表述为“不会误触发”

2. **收盘价有效性校验**
   - 15:01 补检前，应确认获取到的是“当日有效收盘数据”或明确的“闭市后价格”
   - 如果无法确认，应延后到更晚时间或放弃本次补检

3. **补检任务的时区必须统一**
   - 所有 CronTrigger 示例统一写明 `timezone=ZoneInfo("Asia/Shanghai")`
   - 删除不带 timezone 的示例，避免实现者照抄错误版本

#### 建议结论

15:01 补检可以保留，但必须在设计上从“经验性假设”升级为“带前提校验的补检策略”。

### 4. Telegram、watchlist、现有告警开关之间的关系未定义清楚（Important）

#### 现状

当前设计中，这几个规则存在冲突：

1. 在价格获取部分，先写“创建提醒时要求 ETF 必须在自选中（更简单，推荐）”，后面又写“推荐方案是不限制必须在自选中”
2. 前端部分写“Telegram 未配置时，整个到价提醒区域显示提示‘请先配置 Telegram’”
3. 边界情况中又写“Telegram 未配置时，提醒正常检查和触发，只是通知发送跳过”

这些定义互相冲突，会直接影响前后端实现。

#### 风险

如果不明确，后续实现中可能出现：

- 前端禁止创建，后端却允许存在和触发
- 后端错误复用现有 signal alert 的总开关/资格判断
- 用户删除自选后，到价提醒行为前后不一致

#### 修订建议

在设计文档中新增一张“资格矩阵”表，明确以下规则：

| 场景 | 是否允许 | 说明 |
|------|---------|------|
| 无 Telegram 配置时创建提醒 | 允许 / 不允许（二选一） | 必须明确 |
| 无 Telegram 配置时查看/管理提醒 | 是否允许 | 必须明确 |
| 无 Telegram 配置时参与调度检查 | 是否允许 | 必须明确 |
| ETF 不在 watchlist 时创建提醒 | 是否允许 | 必须明确 |
| 删除 watchlist 后提醒是否继续有效 | 是否继续 | 必须明确 |
| 是否受 signal alert 总开关影响 | 是 / 否 | 必须明确 |

#### 建议结论

建议第一版采用如下规则：

- **提醒与 watchlist 解耦**：允许创建不在自选中的提醒；删除自选后提醒仍有效
- **提醒与 signal alert 开关解耦**：不受现有技术信号总开关影响
- **Telegram 未配置时允许管理，但不建议允许创建**：若允许创建，应在 UI 中明确提示“将不会收到通知”

若希望降低第一版复杂度，更简单的方案是：
- **要求 Telegram 已配置后才能创建提醒**
- 但已存在提醒仍可查看/删除

### 5. API 语义不够自洽（Important）

#### 现状

当前 API 设计允许在 `PUT /api/v1/price-alerts/{id}` 中通过：

```json
{
  "is_triggered": false
}
```

来表示“重新启用”。

但设计未定义：

- 传 `is_triggered: true` 是否允许
- 修改价格/方向时，是否允许顺便重新启用
- 重新启用后是否重新校验当前价格

#### 风险

这种“复用状态字段表达动作”的方式容易导致接口语义模糊，也不利于前端实现和后端校验。

#### 修订建议

优先考虑以下两种方案之一：

**方案 A：保持 PUT，但收紧规则**
- `is_triggered` 仅允许从 `true -> false`
- `false -> true` 不允许客户端直接设置
- 文档明确：重新启用时自动清空 `triggered_at / triggered_price`
- 明确重新启用前需要重新校验当前价是否已满足条件

**方案 B：新增显式动作接口（更清晰）**
- 例如：`POST /api/v1/price-alerts/{id}/reactivate`
- 普通 `PUT` 只负责修改价格/备注/方向
- “重新启用”成为单独动作，语义更清楚

#### 建议结论

如果追求第一版接口简单，可保留 PUT 方案，但文档必须补齐行为约束；若追求长期可维护性，建议拆成独立 `reactivate` 动作。

---

### 6. 输入校验和 HTML 消息安全性不足（Important）

#### 现状

设计文档中写明“ETF 代码校验：不校验，前端入口已保证有效性，YAGNI”。

但项目规范要求：

- 所有用户输入必须验证和清理
- 前端输出/通知时要注意 XSS/HTML 风险

此外，通知消息设计中使用 `parse_mode="HTML"`，但消息模板里直接拼接 `etf_name` 和 `note`。

#### 风险

- 后端若完全信任前端传入的 `etf_code` / `etf_name` / `note`，不符合项目安全规范
- 若 `note` 含 HTML 特殊字符，可能破坏 Telegram 消息格式

#### 修订建议

文档应明确：

1. **ETF code 至少做格式校验**
   - 例如只允许 6 位数字（若当前系统 ETF 编码规则如此）

2. **ETF name 尽量由服务端可信数据补全**
   - 若暂时保留前端传入，也应在服务端做长度/字符清理

3. **HTML 消息内容统一转义**
   - 至少对 `etf_name` 和 `note` 做 HTML escape

#### 建议结论

删除“ETF 代码不校验”的设计决策，改成“第一版做最小必要校验”。

---

### 7. 一次性提醒缺少“触发成功但通知失败”的补偿语义（Important）

#### 现状

当前流程中，提醒一旦命中条件就会先被标记为 `is_triggered = true`，然后再发送 Telegram 通知。

#### 风险

若出现以下情况：

- Telegram 配置缺失
- Telegram 接口失败
- 网络异常

则提醒会变成“已触发”，但用户没有收到消息。对一次性提醒来说，这属于高感知问题。

#### 修订建议

第一版至少应在设计中明确其中一种处理方案：

**最小可行方案**
- 保持 `is_triggered=true` 逻辑不变
- 但新增发送结果记录，如 `notification_status` / `notified_at` / `notification_error`
- 在管理页显示“已触发但通知失败”状态

**更进一步方案**
- 允许对发送失败的提醒重试通知
- 或在管理页支持“重新启用并再次等待触发”

#### 建议结论

第一版不一定要实现完整重试机制，但要把“触发成功、通知失败”的状态设计出来，否则会出现静默丢提醒。

## 三、次优先级修订项（Minor / Should Fix）

### 8. 索引设计与查询模式不完全匹配

#### 现状

当前仅设计了：

- `(user_id, is_triggered)`
- `(is_triggered)`

但文档中的实际查询还包括：

- `WHERE user_id = ? ORDER BY is_triggered, created_at DESC`
- 已触发记录按 `triggered_at` 清理
- 活跃提醒按 ETF 代码聚合取价

#### 修订建议

可考虑将索引调整为更贴近查询模式的组合，例如：

- `(user_id, is_triggered, created_at)`
- `(is_triggered, triggered_at)`
- `(is_triggered, etf_code)`

不要求第一版就做到最优，但至少要在文档中说明当前索引是“初版索引”，后续可按查询慢日志调整。

---

### 9. 时间契约没有单独收敛说明

#### 现状

当前文档中：

- 数据库字段使用 `datetime.utcnow()`
- API 响应示例使用 ISO UTC 字符串（`...Z`）
- Telegram 时间展示用 `strftime('%Y-%m-%d %H:%M')`
- 调度器示例有的带 `Asia/Shanghai`，有的没带

这说明“数据库、API、展示层”的时间契约尚未在文档里统一说明。

#### 修订建议

新增一个“时间处理约定”小节，明确：

- 数据库存 UTC
- 调度器使用 `Asia/Shanghai`
- 交易时间判断使用 `Asia/Shanghai`
- API 返回时间格式采用何种标准（如果项目已有统一约定，应与之保持一致）
- 前端展示和 Telegram 消息统一使用北京时间字符串

---

### 10. `float + epsilon` 可用，但需收窄表述

#### 现状

文档目前选择 `float + 0.0001` 作为价格比较策略。

#### 风险

对 ETF 三位小数场景，第一版基本可用；但长期看，浮点比较在边界上仍不够稳健。

#### 修订建议

第一版可以保留该方案，但文档中应加一句：

- 比较前统一保留 3 位小数，减少浮点误差
- 后续若该能力扩展到更广泛品类，再考虑使用 `Decimal` 或整数化存储

---

### 11. V1 范围略宽，建议标注核心与可选项

#### 现状

当前设计已包含：

- 后端模型 / 服务 / API
- 调度器集成
- 15:01 收盘补检
- 30 天自动清理
- 详情页创建入口
- 设置页管理列表
- 重新启用
- 每日摘要可选集成

虽然其中部分已标明“可选”，但整体对第一版来说仍稍宽。

#### 修订建议

在文档中明确分层：

**V1 核心范围**
- 创建提醒
- 查看提醒
- 删除提醒
- 自动检查并触发通知

**V1.1 / 后续优化**
- 重新启用
- 每日摘要中的提醒状态汇总
- 通知失败重试
- 更丰富的过滤与批量操作

这样实现时更容易控制优先级，避免核心正确性问题未解决就扩范围。

## 四、推荐的修订后决策（建议版）

为便于后续真正落地，建议将设计修订为以下更稳的版本：

### 核心产品语义
- 第一版采用“**到达目标价提醒**”，不强调“穿越”
- 当提醒创建/重新启用时，如果当前价格已经满足条件，则拒绝保存并提示用户

### 调度与幂等
- 价格提醒检查与通知发送必须具备原子占有逻辑
- 只有成功把提醒从活跃状态改为已触发状态的执行器，才能发送通知
- 调度任务统一使用中国时区，避免本地时区漂移

### 资格规则
- 到价提醒不依赖 watchlist
- 到价提醒不受 signal alert 总开关控制
- Telegram 未配置时：
  - 若追求简单，禁止创建但允许查看/删除现有提醒
  - 若追求灵活，允许创建但必须明确提示“不会收到通知”

### 安全与校验
- 后端必须校验 `etf_code` 和 `target_price`
- `note` 长度和内容必须清理
- Telegram HTML 消息输出前必须 escape 用户可控字段

### 第一版功能范围
- 只做 CRUD + 调度检查 + Telegram 通知 + 管理列表
- 将“重新启用”“每日摘要汇总”明确为后续增强项，除非实现时证明成本极低且不会影响核心可靠性

---

## 五、建议的后续动作

正式进入开发前，建议按以下顺序推进：

1. **先修订原设计文档** `docs/design/2026-03-05-price-alert-design.md`
   - 统一语义
   - 删除互相冲突的规则描述
   - 补全并发、交易日、时间契约、安全约束

2. **再输出一版实施文档**
   - 放在 `docs/implementation/` 或 `docs/planning/`
   - 明确第一版范围、验收标准、测试方案

3. **实现前再次做一次小型设计复审**
   - 重点复核：是否还存在“立即触发”“重复通知”“触发后丢消息”三类问题

## 六、实施前检查清单

在开始编码前，至少确认以下问题都已被设计文档明确回答：

- [ ] 到价提醒到底是“达到”还是“穿越”？
- [ ] 当前价已满足条件时，创建/重新启用是否允许？
- [ ] 并发下如何保证同一提醒只通知一次？
- [ ] 15:01 补检如何确认拿到的是有效收盘价？
